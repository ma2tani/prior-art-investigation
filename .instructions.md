# Prior Art Investigation Agent

Quickly identify if your feature concept already exists, what it's called, and what OSS solutions are available — **before** you code.

## What This Agent Does

1. **Names Your Concept** — Discovers the real technical term (e.g., "auto-refresh cache" → "Cache-Aside pattern")
2. **Maps Risk** — Identifies what could go wrong (architectural, performance, adoption)
3. **Lists OSS Options** — Shows existing solutions you could adapt or build on

## When to Use

### Phase 1: Requirements (Quick Check)
- ✅ Have a problem statement?
- ✅ Before moving to design?
- **Use**: `@prior-art-investigation minimal` — concept name + OSS list + risk flags

### Phase 2: Design (Deep Investigation)
- ✅ Architecture decisions needed?
- ✅ New subsystem design?
- **Use**: `@prior-art-investigation full` — research lineage + OSS matrix + failure modes

### Phase 3: Unsure Which
- ✅ First time here?
- **Use**: `@prior-art-investigation selector` — auto-routes to minimal or full

## Quick Examples

### Example 1: Requirements Phase
```
User: "@prior-art-investigation minimal I need to implement rate limiting for our API"

Agent:
- Real concept: "Token Bucket" or "Sliding Window Counter"
- Q1 First Principles: Is the problem limiting requests or protecting service quality?
- Q6 Risks:
  • Assumption: Clients handle errors with retry logic
  • Risk: Thresholds too strict may block legitimate users
  • Verify: Measure traffic patterns before setting limits
- Recommend FULL investigation for Design phase
```

### Example 2: Design Phase
```
User: "@prior-art-investigation full We're designing consistency guarantees for a distributed system"

Agent:
- Q1-Q7 complete investigation
- Real concepts: "Eventual Consistency", "CAP Theorem", "CRDT"
- Existing OSS: etcd, Apache ZooKeeper, Consul
- Architecture options: Strong consistency vs eventual consistency
- Recommend: etcd for strong consistency, CRDT pattern for high availability

Provides full research template saved to .kiro/specs/[feature]/docs/research.md
```

## Commands

| Command | Effect |
|---------|--------|
| `@prior-art-investigation selector` | Auto-detect phase, run appropriate prompt |
| `@prior-art-investigation minimal` | Q1 + Q6 only — concept name, OSS list, risk flags |
| `@prior-art-investigation full` | Q1-Q7 complete — research lineage, OSS matrix, tradeoffs, failure modes |

## Outputs

All investigations produce:
- ✅ Concept name confirmed + alternative names
- ✅ OSS evaluation matrix — license, maintainer, update frequency, data prep effort, source links (mandatory)
- ✅ Risk map (assumptions → potential failures)
- ✅ Research template (save to `.kiro/specs/` for Kiro SDD workflows)

**Source linking rule**: every concept, paper, OSS project, and failure pattern in the output
must include a Markdown link (arXiv / GitHub / official docs). Facts without sources are not accepted.

## Integration

### Standalone (VS Code)
```
@prior-art-investigation minimal
```

### With Kiro SDD
```bash
# Phase 1 (Requirements)
/kiro-spec-requirements FEATURE-NAME
# → Runs minimal investigation

# Phase 2 (Design)
/kiro-spec-design FEATURE-NAME
# → Runs full investigation, saves to .kiro/specs/FEATURE/docs/research.md
```

### Hook Formats: Kiro IDE vs VS Code

**Kiro IDE hooks** (`.json` with `enabled` flag):
- Trigger: Command-specific (`after_kiro_spec_requirements`, `after_kiro_spec_design`)
- Location: `.kiro/hooks/prior-art-*.json`
- Control: Edit `"enabled": true/false`

**VS Code Copilot Chat hooks** (`.kiro.hook` format):
- Trigger: `agentStop` (session end) with git diff detection
- Location: `.kiro/hooks/prior-art-*.kiro.hook`  
- Control: File rename (`.disabled` extension to disable)

Use **Kiro IDE** hooks if using Kiro SDD workflow; use **VS Code** hooks if using VS Code Copilot Chat.

## Token Optimization: Control Hook Execution

Prior art investigations consume tokens. You can disable hooks for contexts where investigation is unnecessary:

**To disable**:
1. Edit `.kiro/hooks/prior-art-requirements.json`: change `"enabled": true` to `"enabled": false`
2. Edit `.kiro/hooks/prior-art-design.json`: same change

**Disable when**: maintaining existing codebase, refactoring, well-known patterns  
**Keep enabled**: greenfield projects, major architecture decisions, tech selection
