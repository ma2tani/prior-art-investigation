# 先行技術・既発明調査

> **目的**: 要件・設計・コードを書く前に、その問題が既知の名前・確立されたパターン・既存実装を持っているかを確認する。  
> **スタンドアロン対応**: Kiro SDD不要。任意の開発ワークフローで使用可能。

---

## 実行タイミング

各フェーズの開始時にこのプロンプトを実行する:

| フェーズ | 適用する問い | 深さ |
|---------|------------|------|
| 要件定義 | Q1（第一原理）+ Q6（反転） | クイック — 2問のみ |
| 設計 | 7つ全問 | Phase A + B 全調査 |
| タスク設計 | Q7（So What） | クイック — タスクが調査結果を反映しているか確認 |

---

## Step 1: コンテキストの読み込み

開始前に以下を確認する:
1. 機能または問題の説明（1〜2文）
2. 関わる技術レイヤー: ML/AI・バックエンド・フロントエンド・データ・インフラ・セキュリティ など
3. 要件や設計メモですでに挙がっている概念名（あれば）

---

## Step 2: 設計調査の7つの問いを適用する

順番に問いかけ、各問いへの答えを記録してから次に進む。

| # | 問い | 適用フェーズ |
|---|-----|------------|
| **Q1. 第一原理** | 「問題の枠組みは正しいか？解くべき問題を解いているか？」 | 要件、設計 |
| **Q2. 帰無仮説** | 「なぜこのアプローチは既に普及していないのか？自明なら主流になっているはず — なぜそうなっていないか？」 | 設計 |
| **Q3. 失敗例から始める** | 「このアプローチで失敗した人は何人いるか？どのように失敗したか？」 | 設計 |
| **Q4. 専門家を見つける** | 「このドメインを最も深く考えている人は誰か？その言葉はどこにあるか？」 | 設計 |
| **Q5. 一次資料を読む** | 「READMEやブログ記事ではなく、一次資料（論文・RFC・コミットログ・Issue）を読んだか？」 | 設計 |
| **Q6. 反転** | 「これが最悪の形で失敗するとしたら何が原因か？何を確認すべきか？」 | 要件、設計 |
| **Q7. So What** | 「概念名が分かった。これが設計をどう変えるか？」 | タスク |

> **要件クイックチェック**: Q1 + Q6のみ適用。  
> **タスククイックチェック**: Q7のみ適用。

---

## Step 3: Phase A — 概念の特定

### A-1: 概念を命名する

対象の機能・アプローチについて:
1. 既知のアルゴリズム名・パターン名・技術名が存在するか？
2. 最初に発表されたのはいつか？（成熟度の指標）
3. どのドメインに属するか？（以下の表を参照）
4. 同じ問題を指す別名が存在するか？

### A-2: ドメイン別の調査着眼点

| ドメイン | 確認すべき概念例 | 主なリソース |
|--------|--------------|------------|
| ML / AI | Knowledge Distillation・LoRA・RAG・Active Learning・RLHF | arXiv cs.LG・Papers with Code |
| Webバックエンド | CQRS・Event Sourcing・Saga・Circuit Breaker・BFF | Martin Fowlerのbliki・microservices.io |
| Webフロントエンド | Island Architecture・Optimistic UI・PRPLパターン | web.dev・Chrome Developers |
| モバイル | Offline-first・MVVM/MVI・単方向データフロー | Apple Developer・Android Developers |
| インフラ | Blue-Green Deploy・カナリアリリース・GitOps・Cell-based Architecture | CNCF Landscape・AWS Architecture Blog |
| データプラットフォーム | Medallion Architecture・Data Mesh・Lambda/Kappa・CDC | dbtブログ・Databricks Engineering |
| セキュリティ | RBAC/ABAC・Zero Trust・PKCE・CSP | OWASP・NIST SP 800 |

### A-3: リサーチリソース

ドメインに合わせて2〜3つ選ぶ。1〜2本の資料を深く読む方が10本流し読みするより価値がある。

| カテゴリ | URL | 用途 |
|--------|-----|-----|
| ML プレプリント | https://arxiv.org/list/cs.LG/recent | 最新MLアルゴリズム |
| 実装付き論文 | https://paperswithcode.com/ | 論文 + OSS対応表 |
| 論文検索 | https://www.semanticscholar.org/ | 引用数・関連研究 |
| 設計パターン | https://martinfowler.com/ / https://microservices.io/ | バックエンド・DDD・マイクロサービス |
| 業界レーダー | https://www.thoughtworks.com/radar | Adopt/Trial/Assess/Hold分類 |
| クラウドネイティブ | https://landscape.cncf.io/ | インフラパターン |
| フロントエンド | https://web.dev/ | ブラウザ・パフォーマンスパターン |
| セキュリティ | https://owasp.org/ | 脆弱性・セキュリティ標準 |

## Step 4: Phase B — OSS / 実装の探索

Phase Aで概念が存在することを確認した後にのみPhase Bを実行する。

### B-1: 調査順序
1. まずプロジェクト内部の再利用可能モジュールを確認
2. エコシステムの公式パッケージ（npm・PyPI・crates.io・Hexなど）
3. 業界標準OSS
4. 参照実装（採用なし — レビューのみ）

### B-2: ライセンスTier分類

| Tier | ライセンス例 | 判断 |
|------|-----------|-----|
| **Tier 1** ✅ | MIT・BSD-2/3・Apache-2.0・ISC | 自由に採用可 |
| **Tier 2** ⚠️ | LGPL-2.1/3.0・MPL-2.0・EPL-2.0 | 利用形態を要確認 |
| **Tier 3** 🔴 | GPL-2.0/3.0・AGPL-3.0 | 法的レビュー必要 |
| **Tier 4** ❌ | SSPL・Commons Clause・BSL・CC-BY-NC | 採用禁止 |

### B-3: Build vs Use の判断

| 判断基準 | 既存を採用 | 自前実装 |
|--------|---------|--------|
| 機能適合度 | 要件の80%以上をカバー | 60%未満またはフォーク必要 |
| ライセンス | Tier 1 | Tier 3以上 |
| 最終コミット | 12ヶ月以内 | 2年以上前 |
| 戦略的差別化 | 非コア（ユーティリティ/インフラ） | プロダクトのコアバリュー |
| 自前実装量 | 200行超 | 50行以下 |

---

## Step 5: 調査結果の記録

`templates/research.md`（またはプロジェクトの既存`research.md`）に以下を記録する:

- 概念名と一次資料のURL
- 発表年と成熟度
- 設計への影響（Q7の回答）
- OSSの候補とSPDXライセンス識別子
- 判断: 採用 / 自前実装 / 該当なし

**概念名が見つからなかった場合**: 「概念名なし — X と Y の新規組み合わせ」と根拠を添えて記載。  
**概念名は見つかったがOSSとして採用しない場合**: 自前実装が差別化になる理由を説明する。
