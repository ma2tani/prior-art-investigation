# 調査フレームワーク Q1〜Q8 解説

Prior Art Investigation の `full` モードは8つの質問（Q1〜Q8）で構成されます。  
各質問の**目的・何を問うか・期待されるアウトプット**を以下に説明します。

> `minimal` モードは Q1 + Q6 のみ使用します。

---

## Q1: 第一原理（First Principles）

**目的**: 「本当に解くべき問題か」を確認する。問題の表面的な症状ではなく、根本を問う。

**問い**:
- この問題の定義は正しいか？
- 表面的な症状ではなく、根本的な原因を解こうとしているか？
- 専門用語を除いた最もシンプルな言い方は何か？

**サンプル**:

> 入力: 「API のレスポンスが遅い」  
> Q1 の洞察: 「問題はレスポンス速度ではなく、データベースクエリの N+1 問題。キャッシュより先にクエリ最適化を検討すべき」

**なぜ重要か**: 間違った問題を完璧に解いても意味がない。Q1 はここで軌道修正する。

---

## Q2: 概念名（Concept Name）

**目的**: この問題・解法が既存の研究・エンジニアリング分野で何と呼ばれているかを特定する。

**問い**:
- この問題には既に名前があるか？
- どの確立されたパターンやパラダイムが適用されるか？
- 研究の発展の流れは？（年代別の主要論文）

**サンプル**:

| 年 | 論文 | 重要な洞察 |
|----|------|----------|
| 2015 | [Hinton ら「Distilling the Knowledge in a Neural Network」](https://arxiv.org/abs/1503.02531) | 温度付きソフトマックスで知識転移が可能 |
| 2019 | [Sanh ら「DistilBERT」](https://arxiv.org/abs/1910.01108) | BERT スケールの蒸留が実用的 |
| 2023 | [Li ら「Distilling Step-by-Step」](https://arxiv.org/abs/2212.10071) | LLM の推論プロセス自体を蒸留可能 |

**なぜ重要か**: 概念名が分かれば、既存のドキュメント・OSS・失敗事例を検索できる。ゼロから発明しなくて済む。

---

## Q3: 技術的選択肢（Technical Options）

**目的**: 解法のアプローチや手法として何があるかを網羅する。

**問い**:
- どのアルゴリズム・アーキテクチャパターンが存在するか？
- それぞれのトレードオフは？

**サンプル**:

> 「分散キャッシュ」の選択肢:
> - Cache-Aside パターン（アプリが制御、柔軟）
> - Write-Through（書き込み時に同期、整合性高い）
> - Read-Through（ライブラリが制御、コードがシンプル）
> - Event-Driven Invalidation（変更イベントで無効化）

---

## Q4: OSS エコシステム（OSS Ecosystem）

**目的**: 既存の OSS で既に解かれているか確認し、ゼロから作る前に評価する。

**出力形式**（必須カラム）:

| ツール | ライセンス | メンテナー | 更新頻度 | データ準備工数 | 向いている用途 | ソース |
|------|----------|---------|--------|------------|------------|------|
| Redis | BSD-3 | Redis Ltd（組織） | 活発（毎週） | 低 | 汎用キャッシュ・セッション管理 | [GitHub](https://github.com/redis/redis) |
| Dragonfly | BSL-1.1 | Dragonfly DB（組織） | 活発（毎月） | 低 | Redis 互換・高スループット | [GitHub](https://github.com/dragonflydb/dragonfly) |

**なぜ重要か**: OSS を使えばメンテナンス・セキュリティパッチ・ドキュメントをコミュニティに任せられる。採用コストと運用コストのトレードオフを確認する。

---

## Q5: アーキテクチャ選択（Architecture Choice）

**目的**: Q3〜Q4 の情報をもとに「何を、どう使うか」の推奨を出す。

**問い**:
- どの OSS やアプローチを採用すべきか、その理由は？
- ゼロから作るか、既存を採用するかのトレードオフは？

**サンプル**:

> 「Knowledge Distillation の場合: LLaMA-Factory を採用することを推奨。理由は（1）本番対応済みのパイプラインが揃っている、（2）Apache-2.0 ライセンスで商用利用可能、（3）HuggingFace との統合がネイティブ。一方、独自アーキテクチャに合わせた蒸留が必要な場合は Hugging Face transformers を直接使用する方が柔軟性が高い。」

---

## Q6: 反転（Inversion）

**目的**: 「うまくいくシナリオ」ではなく「失敗するシナリオ」から考える。

**問い**:
- 6ヶ月後にこれが派手に失敗するとしたら、何が原因か？
- どの前提が間違っている可能性があるか？
- 進める**前に**何を検証すべきか？

**サンプル**:

> 「分散キャッシュが失敗するシナリオ:
> - キャッシュ無効化のタイミングが不整合 → 古いデータが表示される
> - キャッシュサイズの見積もりが甘く OOM でサービスダウン
> - Write-Through が遅すぎてレイテンシが改善しないどころか悪化
> 
> 検証すべきこと: 本番トラフィックのキャッシュヒット率を事前にシミュレーション」

**なぜ重要か**: `minimal` モードでも Q6 は必ず実行される。設計前の最も重要なリスクチェック。

---

## Q7: 次のアクション（Next Steps）

**目的**: 調査結果を「次に何をすべきか」の具体的なアクションに変換する。

**問い**:
- 前進するための優先度付きアクションは？

**サンプル**:

> 1. OSS で PoC を作り、実際のトラフィックでキャッシュヒット率を計測（1週間）
> 2. ライセンス確認（BSL-1.1 の場合、商用利用制限を法務に確認）
> 3. 本番環境のメモリ上限からキャッシュサイズ上限を算出
> 4. キャッシュ無効化イベントのテスト戦略を設計フェーズで定義

---

## Q8: プラットフォームネイティブ機能（Platform Native Capabilities）

**目的**: 「ターゲットプラットフォーム（VS Code, GitHub, Azure, AWS 等）が既にネイティブに持っている機能」を再発明していないか確認する。

**問い**:
- このプラットフォームは既にこの問題を解く機能を提供しているか？
- 公式ドキュメント・直近6ヶ月の Changelog・ライブラリの GitHub Releases を確認したか？
- 不必要に再発明してしまう機能はないか？

**サンプル**:

> 「VS Code 向けのカスタムフック機能を独自実装しようと考えていたが、Q8 で確認したところ VS Code 1.99（2025年4月）で `UserPromptSubmit` / `PostToolUse` 等のライフサイクルフックが正式にサポートされた。実装は不要であり、`.github/hooks/*.json` に設定を書くだけで同等の動作が得られる。」
>
> ソース: [VS Code Hooks ドキュメント](https://code.visualstudio.com/docs/copilot/customization/hooks)（2026年4月更新）

**なぜ重要か**: プラットフォームのネイティブ機能は自分たちでメンテする必要がない。特に Q8 の調査対象はアーキテクチャ論文（arXiv）ではなく**公式ドキュメントと Changelog** が主要ソースになる点が他の Q と異なる。

## モード別 Q の使われ方

| モード | 使用する Q | 想定時間 | トークン目安 |
|--------|-----------|---------|------------|
| `minimal` | Q1 + Q6 | 5〜10 分 | 〜150 |
| `full` | Q1〜Q8 全部 | 20〜40 分 | 〜500 |
| `selector` | 状況に応じて自動選択 | — | — |

---

## ソースリンクルール

すべての Q の出力において、以下のルールが適用されます：

- 概念・論文・OSS・失敗パターンを言及する場合、**必ず Markdown リンクを付ける**
- リンクのない主張は認められない
- 受け入れられるソース種別: arXiv / GitHub / 公式製品ドキュメント / Changelog & Release Notes / DOI

> ⚠️ **Training Cutoff に注意**: LLM の学習データには時差があります。以下は必ず手動で確認してください。
> - [ ] ターゲットプラットフォームの Changelog（直近6ヶ月）
> - [ ] 公式ドキュメントの最新バージョン
> - [ ] 主要ライブラリの GitHub Releases
