# 先行技術調査エージェント

機能を実装する前に、その概念が既に存在するか、何と呼ばれているか、どの OSS ソリューションが利用可能かを素早く特定します。

## このエージェントの役割

1. **概念に名前を付ける** — 本当の技術用語を発見（例：「キャッシュの自動更新」→「Cache-Aside パターン」）
2. **リスクをマッピング** — 何が失敗する可能性があるか特定（アーキテクチャ、パフォーマンス、採用）
3. **OSS オプションをリスト** — 既存のソリューションで活用できるものを表示

## 使用時期

### フェーズ 1: 要件定義（クイックチェック）
- ✅ 問題を定義している？
- ✅ 設計フェーズに進む前？
- **使用**: `@prior-art-investigation minimal` — 技術名 + OSS リスト + リスクフラグ

### フェーズ 2: 設計（深層調査）
- ✅ アーキテクチャ決定が必要？
- ✅ 新しいサブシステムを設計している？
- **使用**: `@prior-art-investigation full` — 研究系譜 + OSS マトリクス + 失敗モード

### フェーズ 3: 不明な場合
- ✅ 初回利用？
- **使用**: `@prior-art-investigation selector` — minimal か full に自動振り分け

## クイック例

### 例 1: 要件定義フェーズ
```
ユーザー: "@prior-art-investigation minimal API のレート制限を実装したい"

エージェント:
- 実際の概念: 「Token Bucket」または「Sliding Window Counter」
- Q1 第一原理: 問題は「リクエスト数を制限すること」か「サービス品質を保護すること」か？
- Q6 リスク:
  • 仮定: クライアントはエラーをリトライハンドリングしている
  • リスク: 厳しすぎる制限が正規ユーザーをブロックする
  • 検証: トラフィックパターンを計測してから閾値を設定する
- 設計フェーズで FULL 調査を推奨
```

### 例 2: 設計フェーズ
```
ユーザー: "@prior-art-investigation full 分散システムでの整合性保証を設計しています"

エージェント:
- Q1-Q7 完全な調査
- 実際の概念: 「結果整合性」「CAP 定理」「CRDT」
- 既存 OSS: etcd、Apache ZooKeeper、Consul
- アーキテクチャオプション: 強整合性 vs 結果整合性
- 推奨: 用途に応じて etcd（強整合性）または CRDT パターン（高可用性優先）

研究テンプレートを .kiro/specs/[feature]/docs/research.md に保存
```

## コマンド

| コマンド | 効果 |
|---------|------|
| `@prior-art-investigation selector` | フェーズを自動検出して適切なプロンプトを実行 |
| `@prior-art-investigation minimal` | Q1 + Q6 のみ — 技術名・OSSリスト・リスクフラグ |
| `@prior-art-investigation full` | Q1-Q7 全体 — 研究系譜・OSSマトリクス・トレードオフ・失敗モード |

## 出力

すべての調査は以下を生成：
- ✅ 概念名確認 + 代替名
- ✅ OSS 評価マトリクス — ライセンス・メンテナー・更新頻度・データ準備手間・ソースリンク（必須）
- ✅ リスクマップ（仮定 → 潜在的な障害）
- ✅ 研究テンプレート（Kiro SDD ワークフローで `.kiro/specs/` に保存）

**ソースリンクルール**: 出力内のすべての概念・論文・OSS・失敗パターンには必ず Markdown リンク（arXiv / GitHub / 公式ドキュメント）を付けること。リンクのない主張は認めない。

## 統合

### スタンドアロン（VS Code）
```
@prior-art-investigation minimal
```

### Kiro SDD と統合
```bash
# フェーズ 1（要件定義）
/kiro-spec-requirements FEATURE-NAME
# → minimal 調査を実行

# フェーズ 2（設計）
/kiro-spec-design FEATURE-NAME
# → full 調査を実行、.kiro/specs/FEATURE/docs/research.md に保存
```

### フック形式: Kiro IDE vs VS Code

**Kiro IDE フック**（`enabled` フラグ付き `.json`）:
- トリガー: コマンド固有（`after_kiro_spec_requirements`、`after_kiro_spec_design`）
- 場所: `.kiro/hooks/prior-art-*.json`
- 制御: `"enabled": true/false` を編集

**VS Code Copilot Chat フック**（`.kiro.hook` 形式）:
- トリガー: `agentStop`（セッション終了）+ git diff 検知
- 場所: `.kiro/hooks/prior-art-*.kiro.hook`
- 制御: ファイルリネーム（`.disabled` 拡張子を追加して無効化）

Kiro SDD ワークフロー使用時は **Kiro IDE** フック、VS Code Copilot Chat 使用時は **VS Code** フックを使用。

## トークン最適化：フック実行の制御

先行技術調査はトークンを消費します。調査が不要な場面ではフックを無効化できます：

**無効化方法**：
1. `.kiro/hooks/prior-art-requirements.json` を編集： `"enabled": true` を `"enabled": false` に変更
2. `.kiro/hooks/prior-art-design.json` を編集：同じように変更

**無効化すべき場面**: 既存コード保守・リファクタリング・既知パターン  
**有効化すべき場面**: 新規開発・大型アーキテクチャ決定・技術選定

## 主な特徴

🎯 **フェーズ対応** — クイックゲートには minimal を、深い決定には full を実行

🚀 **ゼロ設定** — `@prior-art-investigation` と入力するだけ

📦 **OSS 重視** — 事前にスクリーニングされたオープンソースソリューション

🔄 **Kiro SDD 互換** — 出力を仕様ワークフローに保存

💬 **バイリンガル** — 英語と日本語で動作

## 次のステップ

1. **初回利用？** → `@prior-art-investigation selector` を実行
2. **要件を持っている？** → `@prior-art-investigation minimal [あなたの機能]` を実行
3. **深掘り必要？** → `@prior-art-investigation full [あなたのサブシステム]` を実行
4. **Kiro を使用中？** → 結果は `.kiro/specs/[feature]/docs/research.md` に自動保存
